# Database Migrations

Schema lives in `src/lib/db/schema.ts`. Migrations in `src/lib/db/migrations/`
are generated by drizzle-kit and applied via a custom runner that uses the
Neon HTTP driver.

## Check DB version (read-only)

```bash
npm run migrate:status
```

Prints the current DB version, the target version from the repo, and any
pending migrations. Exits 0 if up to date, 2 if migrations are pending, 1 on
error. Useful in CI to fail a deploy gate if someone forgot to run migrations.

## Apply pending migrations

```bash
npm run migrate
```

Runs `scripts/migrate.ts`, which:

1. Loads `DATABASE_URL` from `.env.local`.
2. Connects via `@neondatabase/serverless` (HTTP — same driver the app uses).
3. Prints the current → target version diff.
4. Applies any SQL files under `src/lib/db/migrations/` that aren't recorded
   in `drizzle.__drizzle_migrations`.
5. Prints the resulting version.

Idempotent — already-applied migrations are skipped automatically. Safe to run
on every deploy whether or not there are pending migrations.

## Versioning model

A "version" here is the `idx` field in `src/lib/db/migrations/meta/_journal.json`
— a zero-based integer that increments with each generated migration.

- **Current version** = `idx` of the latest migration whose `when` timestamp
  matches the most-recent row in `drizzle.__drizzle_migrations`.
- **Target version** = `idx` of the latest entry in `_journal.json`.
- `current == target` → database is at schema head; nothing to do.
- `current < target` → there are N pending migrations; `npm run migrate` applies
  them in order.

## Runtime status check

`GET /api/migrations/status` (admin/super-admin auth required) returns the same
shape as the CLI:

```json
{
  "currentVersion": 5,
  "currentTag": "0005_scheduled_alerts",
  "targetVersion": 6,
  "targetTag": "0006_alerts_notified_at",
  "isCurrent": false,
  "pending": [{ "idx": 6, "tag": "0006_alerts_notified_at", "when": ... }]
}
```

The admin portal also renders an amber banner across the top of every admin
page whenever the DB is behind schema, telling the operator to run
`npm run migrate`.

> Do **not** use `npx drizzle-kit migrate` against Neon. drizzle-kit's migrate
> command doesn't support the HTTP driver and silently no-ops (this is what
> caused the earlier production outage where migrations 0003/0004 appeared to
> have run but hadn't).

## Create a new migration

1. Edit `src/lib/db/schema.ts`.
2. Generate the SQL + snapshot:

   ```bash
   npm run db:generate -- --name <short_name>
   ```
3. Hand-edit the generated `.sql` if you need data migrations or
   idempotency guards (`ON CONFLICT DO NOTHING`, etc.).
4. Apply locally:

   ```bash
   npm run migrate
   ```
5. Commit both the `.sql` file and the updated `meta/` snapshot.

## Rollback

There is no auto-generated down-migration. If you need to revert:

1. Write a new forward-migration that undoes the change.
2. Or, if you caught it immediately and no data was written yet, delete the
   row from `drizzle.__drizzle_migrations` and run the inverse SQL by hand.

## Production deploy

After `git pull`:

```bash
npm run build
npm run migrate
# restart Next.js process (pm2 restart candy, systemctl restart colorcrush, etc.)
```
